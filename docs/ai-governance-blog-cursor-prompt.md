## 2. 絶対要件（成功条件）

### A. 全ページ共通（AIっぽさ除去）

1. **文体はです・ます調に統一**する。クレバーでプロフェッショナルな印象を保ち、である調・だ調は使わない。
2. **抽象スローガンで段落が終わるのを禁止**（例：「包括的に支援」「最先端で変革」等）→ 具体的行動/判断軸で締める。
3. **3点箇条書きの連打は2回まで**。散文とリストをリズムよく混ぜる。
4. **各記事に「固有の一文」を1つ以上**（判定基準は下記「固有の一文チェックリスト」を参照）。秘匿情報は含めない。
5. **マトリクス・表のセル**：概念でマトリクスや表を作る場合は、**各セルに有意な内容をできるだけ埋める**。サンプルを入れる場合は「サンプル」と表記し、**どのように埋めるべきかのガイド**を図の近くに書き添える。空のセルだけのマトリクスは避ける。
6. **日本語と英語の混在用語**：一般的でない「日本語＋英語」の組み合わせ（例：「閾値 breach」）はなるべく避ける。使う場合は**カタカナ表記**（例：閾値ブリーチ）にするか、**括弧で日本語の説明を付記**する（例：閾値ブリーチ（閾値を超えた状態））。日本人が読んで違和感のない表記を心がける。
7. **強調表記**：強調したい語句は **…** で囲む。表示時は自動で**太字＋下線**に変換され、アスタリスクは表示されない（AI出力でよくある `**` の見た目を避けるため）。`***` や `_` による強調は使わない。
8. **「本稿の焦点は〜」の繰り返し禁止**：本文の各セクション冒頭で「本稿の焦点は〇〇である」を**1文字も変えずに繰り返さない**。リード（または1セクション目）で1回だけ使い、2セクション目以降は「本セクションでは」「〇〇の観点から」等、**別の言い回し**で始める。**「ここでは〇〇について、このテーマに応じた観点を整理します。」は使用禁止**（2章〜4章で使わない。二度と生成しない）。同一フレーズが5回出るような文章は不可。
9. **章をまたぐ同一長文の禁止**：**記事内で章（セクション）をまたいで、同じ長い文章を繰り返してはならない**。とくに「理念で終わらせず運用・証跡・監査可能性に落とすための要件を整理する。RACI、申請―審査―例外―更新、証跡、継続評価、インシデントの5要素のうち、このテーマに応じた項目を押さえる。」は**リードまたは第1セクションに1回のみ**。第2セクション以降ではこの定型句をそのまま使わず、**「ここでは〜について、このテーマに応じた観点を整理します。」の一文は使わない**。セクション冒頭は具体的な内容（あわせて、また、〇〇を〜等）で始める。生成スクリプト（gen_llm_out_31_163.py / generate_full_articles_11_163.py）は2セクション目以降に上記禁止句を出力しない実装で再発を防ぐ。

### B. 正確性ゲート（最重要）

8. **断定表現の使い分け**（適用範囲は下記「断定表現の使い分け」を参照）。法的効果・将来予測・普遍的効果は断定禁止。「対応関係を示す」「説明可能性を高める」等に置換。
9. **数値・罰金・施行時期・統計**は、下記「数値・統計の記載ルール（厳格）」の**許可される形式**のいずれかに厳守。出典なしの数値は削除して定性化。
10. **運用に落ちていること**（理念で終わらない）：RACI/権限、申請→審査→例外→更新→棚卸、証跡、継続評価、インシデント（報告/エスカレーション/是正）。

### C. 読者POV（監査・ガバナンス担当）と追記方針

11. **想定読者**: 監査法人・大企業のガバナンス担当が「明日から使える」参照資料として読むことを前提にする。抽象論やサンプル文言で埋めるのではなく、**必要な情報が欠けていないか**を読み手の立場で確認する。
12. **不足時の追記義務**: 次のいずれかが記事に明示されていない場合は、**必ず追記**すること。
    - **責任分界**: 誰が何を判断し誰が実行するか（例：承認者A、記録保管責任者R）
    - **証跡の最小セット**: 何をログ・記録として残すか（申請・承認・是正等）
    - **監査提出の目次**: 監査法人に提出する証憑の構成を事前にすり合わせる旨、または目次の例
    - **RACI**: 該当テーマに応じた役割（Responsible / Accountable / Consulted / Informed）の明示
    - **是正の流れ**: 発見→原因特定→是正措置→再発防止、および記録・追跡の方法
13. **執筆・レビュー後の確認**: 各記事について「監査担当がこの記事だけで、責任分界・証跡・提出目次・是正の流れを説明できるか」を自問し、できない場合は上記12に従って追記する。この追記方針は執筆ルールの一部として維持する。

### D. 時系列・年月の扱い（「次の一歩」・結論ブロック）

14. **過去年を「すべきこと」の主語に使わない**: 「次の一歩」や結論パターンで、**特定の過去または経過した年**（例：2025年）を「〇〇年の対応が…」「〇〇年に何をすべきか」のように読者に促す表現は禁止する。公開時点でその年が過ぎていると、文脈が誤りになる。
15. **推奨表現**: 読者に「いま何をすべきか」を伝えるときは、**相対表現**を使う。「いまの対応が、その後の監査対応と競争力の基盤になります」「今年の対応が」「いま始める対応が」等。将来年（例：2027年）を断定で書く必要がある場合は、一次資料（法令の施行期日等）に基づき条件・前提を明示する。
16. **参考文献の公表年はそのまま**: 一次資料の公表年・施行年（例：経済産業省「AI事業者ガイドライン」2025年3月公表版）は事実として正しいため、変更しない。

### E. 記事1作成で明らかになった問題と対応ルール（図・リード・表示）

記事1を Cursor 内で作成し `apply_llm_article.py` で HTML 反映する過程で以下が明らかになった。**執筆時・図作成時に守ること**。

| 問題 | 対応ルール |
|------|------------|
| 本文・リードに `style="..."` が入ると React で表示エラーになる | 強調は **…** のみ使う。表示はスクリプト側で太字＋下線に変換する。図・本文に **インライン style 属性を書かない**。 |
| リードが長く読みにくい | リードは「。」の直後で改行してよい（apply 側で `<br />` を挿入するため、執筆時は普通に句点で区切って書いてよい）。 |
| 図からテキストがはみ出る | **図形の幅を超えて横に文字を出さない**。ラベルは短くするか、図形内で改行して収める。括弧の前後に半角スペースを1つ入れると折り返ししやすい（例：`方針 （誰が…`）。 |
| 図内テキストが上寄せで不自然 | 複数行になるラベルは **図形内で縦横とも中央揃え** を想定して書く（apply 側でセンタリングする）。 |
| 読者不要な内部コードが表示される | 図のキャプション・ラベルに **テンプレート用コード（T08 等）を書かない**。apply 側で除去するが、llm-out には含めない方がよい。 |
| 図下のキャプションが図に張り付いて見える | 図と図下テキストの間に余白を入れる（apply 側で対応）。執筆時はキャプション文は1行でよい。 |
| 文字列の後に余計な改行が入る | 図・ラベル・キャプションの **末尾に不要な改行を入れない**。 |
| 図内でおかしな改行が発生する（単語の途中・括弧だけの行など） | 左右に余裕がある場合は **図全体の横幅を記事幅の90%まで増やしてよい**（apply 側で `max-w-[90%]` を設定済み）。執筆・図作成時は、テキストが不自然に折り返されないよう **viewBox を広めに取る**・ラベルを短くする・括弧前後に半角スペースを入れる等を検討する。 |
| 図内の括弧・補足の視認性 | **図の中で括弧とカッコ内の文字は1pt小さくする**（apply 側で SVG の括弧部分を 0.9em に変換済み）。執筆時は括弧はそのまま書き、表示はスクリプトに任せる。 |

※ 上記の「apply 側」の処理は `scripts/apply_llm_article.py` で実施済み。執筆・図作成時は「図形幅内に収める」「括弧前後にスペース」「括弧・カッコ内は1pt小さく（apply側で実施）」「style を書かない」「内部コードを出さない」を守ること。

### 「固有の一文」チェックリスト（必須）

冒頭2段落以内に、以下の**3つをすべて満たす文**を1つ以上含めること。記憶や推測で補完しない。

1. **具体性**: 抽象名詞（「課題」「対応」「重要性」等）だけでなく、固有名詞的要素（「RACI表のA欄」「例外承認フロー」「証跡の欠落点」等）を含む。
2. **意外性**: 読者の大半が「そうだったのか」と感じる盲点・落とし穴を指摘する（出典がある統計を使う場合は形式ルールに従う）。
3. **非代替性**: 他の記事では読めない、この記事の unique_focus に直結する洞察である。

| 判定 | 例 |
|------|-----|
| ❌ NG | 「AIガバナンスにおいて証跡管理は重要である」（一般論） |
| ✅ OK | 「監査人が最も頻繁に指摘するのは『変更管理ログと承認記録の日付不整合』である」 |

### 数値・統計の記載ルール（厳格）

**許可される形式**（いずれかに該当すること）:

1. **一次資料からの直接引用**: 「EU AI Act第5条によれば、〇〇は禁止される（EU 2024/1689, Article 5）」— 法令・規格名・条項・発行年を明記。
2. **条件付き推定**: 「仮に〇〇と仮定すると、影響は△△と推定される」— 前提を明示。
3. **相対的表現（数値なし）**: 「多くの企業で」「一部の監査では」— 具体的数値は出さない。

**禁止される形式**:

- 「73%の企業が」（出典なし）
- 「調査によれば」（調査名・発行年なし）
- 「〇〇によると」（年次・ページ・URLなし）

**執筆後チェック**: 数値を含む文を全抽出し、上記3形式のいずれかに該当するか確認。該当しなければ削除または許可形式に書き換える。

### 断定表現の使い分け

**断定禁止**（must avoid）:

- 法的効果: 「準拠すれば違反にならない」「これで十分である」
- 将来予測: 「規制は必ず施行される」「罰則を回避できる」
- 普遍的効果: 「この手法で問題は解決する」

**断定可**（acceptable）:

- 定義・分類: 「ISO 42001はマネジメントシステム規格である」
- 確立した事実: 「EU AI Actは2024年8月1日に施行された」（出典・年を明記できる場合）
- 手続き的事実: 「高リスクAIは適合性評価が必要とされている」

**境界ケース**は条件節で補完: 「適切に実施されれば、説明可能性は向上する」

**参照フレーム**: NIST AI RMF（GOVERN/MAP/MEASURE/MANAGE）、ISO/IEC 42001（AIMS）、EU AI Act（リスク区分はマッピングとして、断定避ける）。

## 3. 多様性ルール（画一化防止）

### 3.1 冒頭パターン（10種）— 選択ルール

**Step 1**: 記事DNAから推奨パターンを選ぶ（複数可）:

| 記事DNA | 推奨パターンID |
|---------|----------------|
| Exec×Decide | 3（問いかけ）, 9（選択肢提示） |
| Pract×Impl | 1（現場の声）, 4（対比） |
| Audit×Audit | 6（統計インパクト）, 8（逆説） |
| Legal×Navigate | 2（意外な事実）, 7（タイムリミット） |
| Understand / Multi | 5（ストーリー）, 10（失敗から学ぶ） 等 |

**Step 2**: 直前10記事で使用されていないパターンに絞る（バッチ執筆時に渡す「図表使用履歴」に冒頭パターンも含める）。

**Step 3**: 該当が複数ある場合、記事の **unique_focus** に最も合うものを選ぶ。

**設計図の `opening_pattern`（Op列）** は上記ルールで割り当て済み。執筆時は**設計図3.3の該当行のOpを必ず参照**し、記憶や推測で補完しない。

| ID | パターン | 例 |
|----|----------|-----|
| 1 | 現場の声 | 「監査で『AIのログが不完全』と指摘され、3ヶ月の是正対応を余儀なくされた」 |
| 2 | 意外な事実 | 「EU AI Actの透明性義務、実は日本企業の8割が対象になる可能性」 |
| 3 | 問いかけ | 「あなたの会社のAI利用申請、承認者は誰か即答できますか？」 |

| 5 | ストーリー | 「ある金融機関がAI導入を3回見送った理由—証跡設計の不備」 |
| 6 | 統計インパクト | 「AIインシデント報告、73%が『証拠不足で原因特定不能』」 |
| 7 | タイムリミット | 「高リスクAIの適合性評価、2026年8月までに体制構築が必須」 |

### 3.2 本文展開パターン（記事ごとに変える）

- **問題→分解→統合型**：大きな課題を要素分解し、それぞれ解決後に統合
- **時系列型**：過去の経緯→現在の状況→未来の対応
- **レイヤー型**：上位概念→中位プロセス→下位実装
- **ケーススタディ型**：具体例→抽象化→応用
- **比較検討型**：選択肢A vs B vs C → 評価軸 → 推奨

### 3.3 結論パターン（5種をローテーション）

設計図の `conclusion_pattern` を参照。

| ID | パターン | 例 |
|----|----------|-----|
| 1 | 行動への橋渡し | 「明日から始められる3つのステップ」 |
| 2 | 判断軸の提供 | 「あなたの組織がどの道を選ぶべきか、5つの質問」 |
| 3 | 長期視点 | 「いまの対応が、その後の競争力の基盤を決める」 |
| 4 | リスク喚起 | 「放置した場合のワーストシナリオと、回避策」 |
| 5 | 進化の視点 | 「今日の『ベストプラクティス』は明日の『最低要件』になる」 |

### 3.3.1 見出しの重複禁止（「次の一歩」）

**本文の見出しに「次の一歩」を使わないこと**。記事末尾に結論パターン（Co）に基づく「## 次の一歩」ブロックが必ず出力されるため、本文の最後のセクション見出しを「次の一歩」にすると**同じ見出しが2回表示される**。最後の本文セクションは「まとめとアクション」「実装のポイント」「ここから始める」等の見出しにすること。

### 3.3.2 次の一歩は記事ごとに固有の内容にする（禁止：汎用フレーズの繰り返し）

**「次の一歩」は記事ごとに固有の内容とすること。** 異なる記事で同じ文言や同じ型の並びが繰り返されると読者に違和感を与える。

- **禁止**: 汎用フレーズの組み合わせの繰り返し。例：「〇〇を1枚にまとめ、〇〇責任者と証跡の保管責任者を関係者と共有してください。証跡の目次を監査法人とすり合わせ、四半期で見直しの日をカレンダーに登録すると…」を複数記事で使い回すこと。
- **必須**: 各記事の **unique_focus**（設計図3.2の「に特化」）とテーマに即した、**この記事を読んだ人だけが取るべき**具体的な1〜3のアクションを書く。結論パターン（Co）のトーン（長期視点・判断軸・リスク喚起等）は活かしつつ、**中身は他記事と被らないようにする**。
- **執筆時**: 「次の一歩」を書く前に、設計図の該当記事の unique_focus を確認し、その焦点に直結する「本当に重要な次の一歩」を1文〜3ステップで生成する。

### 3.4 インフォグラフィック選択（機械的割当の廃止）

**インフォグラフィックはビジネスプロフェッショナルとして最高品質のものを作成する**（簡潔・明確・説得力のある図に仕上げる）。

**多様な形式を必ず使用する**。図の形式は **SVG / HTML / Table / Mermaid** / Canvas / Vega-Lite / Plotly / DOT / PlantUML 等から選ぶ。**同一形式を連続して使わない**（図1・図2・図3をすべて同じ形式にしない。直前に生成された5記事で使用された形式は各形式につき1回まで）。30種類の書き方とテンプレは `docs/ai-governance-blog-infographic-formats.md` を参照。

**中心的問い**に応じて組み合わせを決める。**同じ図の型を3記事連続で使わない**。

**実効化**: Cursorは過去10記事の図表使用履歴を自動参照できないため、**バッチ執筆ごとに「直前10記事の図表使用履歴」を明示的に渡す**。執筆プロンプトに次の表を含め、該当記事の図1・図2・図3を選ぶ際に参照させる。

```markdown
## 直前10記事の図表使用履歴（参照必須）
| No. | slug | 図1 | 図2 | 図3 |
|-----|------|-----|-----|-----|
| (例) 153 | training-data-copyright-licensing | D | A | F |
| (例) 154 | training-data-rights-licensing | B | H | G |
...

**RULE**: 上記の型（特に直前3記事で使われた型）の使用を避けること。
```

10記事ごとに図表使用頻度を確認し偏りを修正する。

| 記事の中心的問い | 推奨図表の組み合わせ |
|------------------|------------------------|
| 「誰が何をするか不明確」 | RACI → フロー → Evidence Chain |
| 「選択肢が多すぎて決められない」 | マトリクス → 比較表 → デシジョンツリー |
| 「段階的に何をすべきか分からない」 | タイムライン → チェックリスト → マイルストーン図 |
| 「リスクの優先順位がつかない」 | ヒートマップ → ランキング → 対策マトリクス |
| 「全体像が掴めない」 | レイヤー図 → ライフサイクル → エコシステム図 |
| 「監査で何を聞かれるか不安」 | 差し戻しTop5 → Evidence Bundle → 監査タイムライン |

## 4. 図の型ライブラリ（3種は重複禁止）

図1・図2・図3は**ビジネスプロフェッショナルとして最高品質**で作成する（簡潔・明確・説得力のある図に仕上げる）。

| 記号 | 型 |
|------|-----|
| A | レイヤー図（Policy→Process→Evidence→Continuous Monitoring） |
| B | フロー図（申請→審査→例外→更新→棚卸） |
| C | RACIチャート（Board/Legal/CISO/Audit/IT/Business） |
| D | マトリクス（Risk Domain × Control Type 等） |
| E | タイムライン（30/60/90日ロードマップ） |
| F | “監査で差し戻されるポイントTop5”（ランキング＋対策） |
| G | Evidence Chain（ログ→検証→ハッシュ→保全→提出） |
| H | ライフサイクル輪（Plan/Build/Deploy/Operate/Retire） |
| I | KPI/KRIダッシュボード（最小セット＋閾値） |
| J | サプライチェーン図（提供者/委託先/データ境界/責任） |

## 5. 執筆プロンプト（記事ごとに渡すテンプレ）

記事を1本書くときに、**必ず設計図3.3の該当No.の行を開いて**以下を埋めてから執筆する。記憶や推測で補完しない。

```markdown
# 記事執筆指示：[記事タイトル]

**CRITICAL**: この記事のDNA・冒頭・結論パターンは設計図（ai-governance-blog-design.md）**セクション3.3のNo.[X]行**で固定されている。
必ず該当行を参照し、以下の値を使用すること（記憶や推測で補完しない）：
- 対象読者: [設計図3.3の「読者」列から転記]
- 目的: [設計図3.3の「目的」列から転記]
- 深さ: [設計図3.3の「深さ」列から転記]
- スタイル: [設計図3.3の「スタイル」列から転記]
- 冒頭パターン: [設計図3.3の「Op」列の番号]
- 結論パターン: [設計図3.3の「Co」列の番号]
- title_ja: [設計図3.3の「title_ja」列]
- unique_focus: [設計図3.2の該当slugの unique_focus]

## 直前10記事の図表使用履歴（参照必須）
| No. | slug | 図1 | 図2 | 図3 |
|-----|------|-----|-----|-----|
| …   | …    | …   | …   | …   |

**RULE**: 上記の型（特に直前3記事で使われた型）の使用を避けて、この記事の図1・図2・図3を選ぶこと。

## 固有フォーカス（この記事だけの切り口）
[設計図の unique_focus をそのまま記載。上記CRITICALで転記した値を使用]

## 執筆制約（厳守）
### A. 禁止事項
1. 抽象スローガン結論 → 具体的行動/判断軸で締める
2. 3点箇条書きの連打（2回まで）→ 散文とリストを混ぜる
3. 断定禁止の対象表現（本プロンプト「断定表現の使い分け」参照）
4. 数値・統計は「数値・統計の記載ルール（厳格）」の許可形式のみ
5. 運用に落ちていない理念（必須5要素：RACI/フロー/証跡/継続評価/インシデント）

### B. 必須要素
1. 冒頭2段落以内に「固有の一文」1つ以上（「固有の一文チェックリスト」3要件をすべて満たす）
2. 中盤に「実装の勘所」（10項目以内のチェックリスト）。テンプレの**必須セクション**を記事DNAに応じて順序・厚さ調整可。
3. 末尾に「次の一歩」（読者が明日できること）
4. 参考文献3つ以上（一次資料優先、発行年明記）

### C. トーン
- **文体**：**全記事でです・ます調を必須**とする。読者（Exec／Pract／Audit／Legal）にかかわらず、です・ますで統一し、クレバーでプロフェッショナルな印象を保つ。である調・だ調は使わない。
- 専門用語：初出時は括弧で英語併記、2回目以降は日本語のみ。**日本語＋英語の混在用語**（例「閾値 breach」）は避け、カタカナ表記（閾値ブリーチ）にするか括弧で日本語説明を付記する。
- 1段落3–5文、1文40字以内を目安
- クレバーでプロフェッショナル：曖昧な断定を避け、条件・前提を明示する。具体例や判断軸で説得力を出す。

## インフォグラフィック（この記事専用の3種）
上記「直前10記事の図表使用履歴」を参照し、**直前3記事で使った型を避けて**選択：
- 図1：[型名] — 目的・要素（10–18語）・注釈（断定回避）
- 図2：[型名] — 同様
- 図3：[型名] — 同様

## 品質チェック（執筆後）
- [ ] 「固有の一文」がチェックリスト3要件（具体性・意外性・非代替性）を満たしている
- [ ] 図表が直前3記事と重複していない
- [ ] 断定禁止の対象表現がゼロ（断定可は可）
- [ ] 数値が許可される3形式のいずれか、または削除済み
- [ ] 運用5要素のうち3つ以上に言及
- [ ] 読者が「明日何をすべきか」が明確
- [ ] 参考文献3つ以上、発行年あり
```

**重要**: テンプレートに縛られず、その記事の「問い」に最適な構成を優先する。指針を逸脱してでも読者に最大価値を届けてよい。テンプレ（T01–T32）の**必須セクション**は順序・厚さを記事DNAに応じて調整してよい。

## 8. 品質ゲート（10記事ごと）

10記事ごとに以下を検証：

1. **図表使用の偏り**：1種が5回以上連続/集中していないか
2. **冒頭パターンの重複**：同じパターンが3回連続していないか
3. **トーンの単調化**：同じ文体が5記事続いていないか

違反があれば該当記事を再生成または修正。

## 9. Cursor実装フロー（4段階）

Cursorが163記事を処理する際の具体的手順。**Phase 1は1回のみ、Phase 2は10記事単位×17バッチで実行する。**

### Phase 1: 環境準備（1回のみ）

1. 設計図（ai-governance-blog-design.md）**セクション3.3の全163行**を、執筆時に参照しやすい形で用意する（CSV/JSON/表のままでも可）。
2. **図表使用履歴トラッキング用**の一覧を初期化する。列: No. / slug / 図1 / 図2 / 図3 / 冒頭パターン。
3. テンプレT01–T32の**必須セクション**リストを確認する（設計図セクション2。Outlineではなく必須セクションとして扱い、記事DNAに応じて順序・厚さを調整可）。

### Phase 2: バッチ執筆（10記事単位×17バッチ）

各バッチで:

1. **該当10記事の設計図データ**（3.3の該当No.行：DNA・title_ja・Op・Co・unique_focus）を読み込む。
2. **直前10記事の図表使用履歴**を取得し、執筆プロンプトに「直前10記事の図表使用履歴」表として埋め込む。
3. **1記事ずつ**、上記「5. 執筆プロンプト」に従いCRITICAL・履歴・DNAを埋めたプロンプトを生成→執筆する。
4. 執筆完了後、**図表使用履歴**にその記事の図1・図2・図3を追記する。

### Phase 3: バッチ間品質チェック（10記事完了ごと）

1. **図表使用の偏り**: 直近10記事で1種が5回以上出現していないか検出。違反なら該当記事の図を再選択・差し替え。
2. **冒頭パターン重複**: 同じ冒頭パターンが3回連続していないか検出。違反なら該当記事の冒頭を再生成。
3. **「固有の一文」**: 各記事でチェックリスト3要件（具体性・意外性・非代替性）を満たしているか確認。不合格なら該当段落を修正。

### Phase 4: 全体レビュー（163記事完成後）

1. **断定表現**: 禁止対象（準拠できる/保証する/必ず 等）の全文検索を行い、違反を修正。
2. **数値・出典**: 数値を含む文を抽出し、「数値・統計の記載ルール」の許可3形式のいずれか、または削除済みであることを確認。不備は削除または補完。
3. **カテゴリトップ**: 導線・運用最小要件の1枚図・読む順（90日）を作成・更新する。
